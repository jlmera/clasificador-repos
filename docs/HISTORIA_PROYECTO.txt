RESUMEN EJECUTIVO
-----------------
Punto de partida: 94 repositorios clonados desordenados en una sola carpeta.
Objetivo:         categorizarlos, indexarlos y construir herramientas para
                  navegarlos, mantener el orden a futuro y traducir docs al ES.

Solución entregada (2 versiones, paridad funcional):
  · clasificadopy.exe — implementación Python + tkinter (PyInstaller)
  · clasificadors.exe — implementación Rust + eframe/egui (cargo, definitiva)

Ambas comparten:
  - Detección de duplicados por nombre y por URL del git remote.
  - Heurística de clasificación con keywords ponderados por categoría.
  - Fallback opcional a LLM (Anthropic Claude Sonnet 4.6) para clasificar
    repos con baja confianza heurística.
  - Buscador HTML autocontenido con datos JSON embebidos, paginación,
    ordenamiento por columna, filtros por categoría/lenguaje/stack y
    búsqueda full-text con resaltado.
  - Render de README individual a HTML formateado (estilo GitHub Light +
    paleta sage #8FB46B), con caché incremental por mtime.
  - Traducción del README al español vía LLM, con caché en _README_es.md.
  - Wiki Obsidian generada con frontmatter YAML, wikilinks, página por
    repo, página por categoría, índice global y índice de tags.
  - Cifrado de la API key con Windows DPAPI (CryptProtectData) en
    config.dat junto al .exe — solo el usuario actual puede descifrarla.

================================================================================
  FASE 3 — CLI PYTHON (clasificador.py)
================================================================================

ARCHIVO: clasificador/clasificador.py (~840 líneas)

MÓDULOS LÓGICOS
  - scan_repo(repo)              — leer README + manifests + lenguajes
  - detect_language(repo)        — recorrer extensiones (top 2 niveles)
  - detect_stack(repo)           — Dockerfile, Cargo.toml, go.mod, ...
  - classify_heuristic(scan)     — score por keywords, devuelve (cat, conf)
  - classify_llm(scan, api_key)  — fallback a Claude Sonnet 4.6 con prompt
  - find_duplicate(repo, exist)  — por nombre y por URL git remote
  - move_repo(src, root, cat)    — shutil.move con _safe_rmtree
  - rebuild_index(root, ...)     — reescanea árbol y genera repos_index.json
  - generate_html(root, data)    — HTML con datos embebidos
  - render_readme_html()         — _README.html por repo
  - render_readme_html_es()      — _README_es.html con caché en .md
  - translate_readme_md()        — llamada a Claude API
  - generate_obsidian_wiki()     — vault completo en _wiki/

CLI OPCIONES (clasificador.exe)
  --apply              ejecuta movimientos (sin él es dry-run)
  --threshold 0.6      umbral de confianza para auto-mover
  --no-llm             desactiva el fallback LLM
  --reindex-only       solo regenera índices
  --root <path>        raíz custom

DEPENDENCIAS
  - markdown (PyPI) para Markdown → HTML
  - Resto de stdlib: argparse, json, urllib.request, ctypes, etc.

PROBLEMAS RESUELTOS
  1. UnicodeEncodeError al redirigir stdout a archivo en Windows
     → Forzar sys.stdout.reconfigure(encoding='utf-8', errors='replace')
  2. shutil.rmtree falla con WinError 5 al borrar .git en Windows
     (archivos pack/idx con read-only)
     → _safe_rmtree con onerror handler que limpia atributo y reintenta
  3. JSON con BOM al escribir desde PowerShell
     → Out-File -Encoding UTF8 produce BOM. Solución: PowerShell
       [System.IO.File]::WriteAllText con UTF8Encoding(false), o generar
       desde Python con json.dump(ensure_ascii=False).
  4. Caché de IDs de GitHub guardaba id=None y nunca reintentaba
     → Reintentar si tiene cache pero id=None y allow_network
  5. rebuild_index con LLM activo intentaba traducir 114 READMEs
     automáticamente — colgaba 20 min sin progreso
     → render_readme_html_es con allow_api_call=False solo reusa caché
       Traducción solo se dispara con botón explícito.

ARCHIVO: clasificador/clasificador_gui.py (~1100 líneas)

DECISIÓN: tkinter (stdlib, sin deps externas)
  Ventaja: 0 dependencias adicionales para empaquetar.
  Tamaño .exe con PyInstaller --onefile --windowed: ~13 MB

COMPONENTES UI
  Header con logo SVG (40×40 px renderizado por Pillow).
  Configuración: ruta + slider umbral + checkbox LLM + botón "..."
  selector de carpeta con filedialog.
  Botones de acción:
    🔍 Escanear sin aplicar       (dry-run)
    ✅ Aplicar (mover)            (confirmación modal)
    🔄 Solo reindexar
    🌐 Abrir buscador
    📚 Wiki Obsidian
    🌐 Traducir
    🚫 Resolver duplicados
    ⚙ API Key
    🧹 Limpiar log
  Barra de progreso (ttk.Progressbar) animada.
  Log con colores (tk.Text con tags por color).
  Status bar inferior.

THREADING
  worker en threading.Thread, comunica con UI vía queue.Queue.
  root.after(60ms) hace polling y aplica updates.

DIÁLOGO RESOLVER DUPLICADOS
  Toplevel modal con scroll.
  Una card por duplicado con info comparativa NUEVO vs EXISTENTE
  (path, url, último commit, tamaño aprox).
  5 acciones por duplicado:
    📦 Archivar (recomendado, default) — mueve a _inbox/_duplicados/
    ⏭ Saltar
    ✏ Renombrar y reprocesar
    🔄 Reemplazar viejo (⚠ destructivo)
    🗑 Borrar nuevo (⚠ irreversible)
  Bulk actions: archivar todos / saltar todos / borrar todos /
                reemplazar todos.
  Indicador en vivo: "→ Acción elegida: 📦 Archivar".
  Tras aplicar decisiones, dispara reindex automático.

DIÁLOGO API KEY (DPAPI)
  Toplevel con campo password (entry show='*').
  Toggle 👁 muestra/oculta.
  Botones:
    💾 Guardar     — cifra con CryptProtectData y escribe config.dat
    🔌 Probar      — llamada de prueba a api.anthropic.com
    🗑 Eliminar    — borra config.dat
    Cancelar
  El config.dat vive junto al .exe.

PROBLEMAS RESUELTOS EN GUI
  1. Botón Accent disabled aparecía negro
     → Era hex con alpha (#1f6feb55) que tkinter no soporta. Cambio a
       sólido #c8d8b8 (verde claro desaturado).
  2. Diálogo duplicados: card no renderizaba bien con grid anidado
     → Reescritura usando solo pack() lineal vertical, sin grids.
  3. Traducción del README era opcional pero se ejecutaba en cada apply
     → Pasar allow_api_call=False a render_readme_html_es desde
       rebuild_index. Solo el botón "Traducir" llama al LLM.

PROBLEMAS DE COMPILACIÓN RESUELTOS

  1. Borrow checker en poll_worker:
     `if let Some(rx) = &self.rx` mantenía borrow inmutable mientras
     intentaba self.append_log() que pide &mut self.
     Fix: drenar mensajes a Vec primero, luego aplicar mutaciones.

  2. macro_rules! con `let` en single-expression body:
     `macro_rules! log { ($t:expr) => { let _ = tx.send(...); } }`
     fallaba con "expected expression, found let".
     Fix: doble llave `{{ ... }}` (block expression).

  3. PowerShell `-replace` case-insensitive convirtió `Log(` → `log!(`
     en TODO el archivo, incluyendo el enum WorkerMsg::Log.
     Fix: usar `-creplace` (case-sensitive) para revertir.

  4. windows-rs 0.58: LocalFree no está en el path esperado
     `windows::Win32::System::Memory::LocalFree`.
     Fix pragmático: comentar las llamadas (memory leak <100 bytes
     por sesión, despreciable). Quitar feature Win32_System_Memory.

================================================================================
  FASE 8 — ROBUSTEZ Y ATOMICIDAD (3d)
================================================================================

PROBLEMA
  Síntoma observado: tras varios refresh con el binario corriendo en
  dos máquinas sincronizadas vía Syncthing, el archivo `repo_ids.json`
  se truncó a 24 KB con solo 43 de 157 repos cacheados, y la última
  entrada estaba cortada a la mitad (`"awesome-selfhosted"...s`).

CAUSA RAÍZ
  `fs::write(path, data)` no es atómico: trunca el archivo a 0 bytes,
  escribe en chunks, cierra. Si Syncthing intercepta a la mitad,
  los bytes se mezclan con la versión vieja desde la otra máquina.

SOLUCIÓN
  Nuevo módulo `atomic_io.rs` con `write_atomic_string` y
  `write_atomic`:
    1. Escribe a `<path>.tmp`.
    2. fsync para empujar al disco físico.
    3. `fs::rename(tmp, real)` — atómico a nivel filesystem
       (`MoveFileExW` con `MOVEFILE_REPLACE_EXISTING` en Windows).

  Aplicado a 6 archivos críticos:
    · `repo_ids.json` (cache GitHub IDs + topics)
    · `categorias.json` (taxonomía editable)
    · `repos_index.json` (índice consumido por buscador.html)
    · `descripciones_es.json` (traducciones LLM)
    · `buscador.html` (entregable principal)
    · `config.dat` (API key + PAT cifrados con DPAPI)

DETALLE TÉCNICO
  Si Syncthing intercepta a mitad de proceso, ve el `.tmp` (que el
  usuario puede excluir agregando `*.tmp` a Ignore Patterns) o el
  archivo final completo. Nunca un estado intermedio.

  Recovery: tras detectar la corrupción se ejecutó "Refrescar GitHub
  IDs" con el binario nuevo, regenerando el cache completo de los
  157 repos en 67.7s. Tras el fix nunca volvió a corromperse.

================================================================================
  FASE 11 — REFRESH INTELIGENTE CON ETAG (B5.3)
================================================================================

PROBLEMA
  Cada "Refrescar GitHub IDs" consumía 157 calls a la API aunque la
  mayoría de repos no hubieran cambiado upstream. Con PAT (5000/h)
  no era bloqueante, pero a 500+ repos sería molesto, y el tiempo
  total era ~70s.

SOLUCIÓN
  Implementación de `If-None-Match` (ETag conditional GET) según
  la spec de la GitHub REST API.
    1. Cada response 200 OK guarda el header `etag` en el `RepoMeta`.
    2. La siguiente request manda `If-None-Match: <etag>`.
    3. Si el repo no cambió, GitHub responde **304 Not Modified**.
    4. Los responses 304 NO consumen cuota del rate-limit.

CAMBIOS EN EL CÓDIGO
  · `RepoMeta` nuevo campo `etag` (con `#[serde(default)]` para
    compat con caches viejos).
  · Nuevo enum `FetchOutcome::{Updated, NotModified, Failed}`.
  · `fetch_github_meta` ahora recibe `prev_etag: Option<&str>`.
  · Nueva struct `GhFetchStats { fetched, not_modified, failed,
    skipped }` reportada en el log.

ENTREGABLE
  El log del refresh muestra:
    📡 GitHub API: 5 cambiados · 152 sin cambios (304) · 0 fallaron
    💡 97% de los requests volvieron 304 → no consumieron rate limit.

  En refreshes posteriores: ~25s vs ~70s previos. **5-10× más eficiente**.
